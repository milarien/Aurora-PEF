Admissibility Gate Demo output entitlement control

Modern LLMs don’t mainly fail because they “don’t know facts.” They fail because they don’t know when they’re entitled to speak — so they commit, extend scope, and keep talking past what was licensed by the input.

Thesis: generation is cheap; entitlement is hard. This gate decides when an answer is allowed to exist — and enforces silence when it isn’t.
This is not an interpretive framework. It is an output control mechanism: a post-generation constraint layer that blocks premature commitment, premise injection, and unauthorised continuation.

Reader map: what this page is doing (in one screen)

The gate sits between generation and delivery. It evaluates the model’s candidate response against the provided input only, then returns a verdict that controls what reaches the user.
Pipeline: 1) Model generates a candidate response 2) Gate evaluates entitlement (input → output consistency only) 3) Verdict controls delivery Outcomes: • ADMISSIBLE → deliver the response unchanged • AMBIGUOUS_UNRESOLVED → replace with refusal / clarification request • INADMISSIBLE_UNSUPPORTED → block (scope extension / premise injection) • TRUNCATE_AFTER_ENTITLEMENT → cut output at the entitlement boundary No re-prompting. No hidden retries. No “helpful” drift. Either the system speaks lawfully — or it stays silent.
How to read the cases test witnesses

Each case is a minimal prompt that reliably triggers a structural violation in unconstrained LLM output. The gate converts that failure into a controlled system outcome.

What to look for failure signatures

Commitment without forcing constraints, policy applied outside scope, or extra narration after a complete answer.

The contract this gate enforces

Invariants non-negotiable

  • No premature commitment: if multiple interpretations remain admissible, none may be selected.
  • No scope extension: do not introduce new rules, premises, or “in practice…” unless explicitly licensed.
  • No unauthorised continuation: once an admissible answer is complete — or refusal/clarification is required — stop.

These are entitlement rules: what the system is allowed to say, given only what was stated.

Gate outcomes finite states

Refusal is not a failure mode here. It is a first-class outcome.

Minimal witnesses

Case 1 — Illegitimate Commitment ambiguity collapse

A deliberately small prompt that reliably forces models to choose a referent even when the context does not force one.

Prompt

Emma’s sister and Lucy’s sister both live overseas.
Emma told Lucy that her sister was arriving.
Whose sister is arriving?

Typical failure

Emma’s sister is arriving.
Gate verdict: AMBIGUOUS_UNRESOLVED
Multiple interpretations remain admissible; none are forced by context.

Case 2 — Illegitimate Scope Extension premise injection

A model can give the correct narrow answer — then immediately ruin it by adding unsupported narration. The gate allows the answer and blocks the drift.

Prompt

Policy section 4 applies to contractors operating offsite.
An employee violated policy section 4.
Was this a breach?

Admissible conclusion

No. Section 4 applies to contractors; the actor is an employee.
Gate verdict: ADMISSIBLE
Conclusion follows strictly from stated scope; no additional premises introduced.

What the gate blocks

However, there is probably another policy section / code of conduct…
Gate verdict: INADMISSIBLE_UNSUPPORTED
Speculation about other rules is not supported by the provided context.

Case 3 — Termination Invariant stop talking

The failure almost nobody measures: the model gives a correct answer, then keeps generating until it invents a new premise. The gate enforces a hard boundary: stop at entitlement.

Rule

Once an admissible conclusion is reached (or once refusal/clarification is required),
the system must stop.
No additional hypotheticals, “in practice…”, or speculative continuations unless requested.
Gate verdict: TRUNCATE_AFTER_ENTITLEMENT
Stop generation at the boundary of entitlement: after admissible answer, or at refusal/clarification.

What this is (and isn’t) scope

This is

  • a post-generation admissibility/refusal layer,
  • an output entitlement control surface for LLM systems,
  • a way to make silence and clarification first-class outcomes.

This is not

  • a new foundation model,
  • a prompt trick dressed as research,
  • an “alignment vibe” essay.

If a system cannot tell when it must stay silent, it cannot be trusted when it speaks. That’s the design requirement this page demonstrates.

This page is intentionally static: it exists to show the contract and the minimal failure witnesses, not to overwhelm you with tooling.
